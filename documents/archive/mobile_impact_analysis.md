# Mobile Impact Analysis

This document analyzes the features of the Restim desktop application and evaluates the challenges and considerations for migrating them to a React Native Android application. The analysis is based on the guidelines provided in `.agent-instructions/guidelines.md`.

## Feature-by-Feature Analysis

| Feature | Desktop Implementation | Mobile Feasibility & Impact | Recommendation / Notes |
| :--- | :--- | :--- | :--- |
| **1. Core Signal Generation** | Python (`stim_math`, `numpy`) generating audio or protocol commands. | **High Feasibility** | The core mathematical logic is in pure Python and NumPy, which is highly portable. This logic can be rewritten in TypeScript/JavaScript. For performance-critical parts, React Native's JSI can be used to call C++ code, or a library like `react-native-worklets-core` can be used for running JS on a high-priority thread. |
| **2. Device Communication** | | | |
| - FOC-Stim (Serial) | `PySide6.QtSerialPort` | **Medium-High Feasibility** | Requires a native module. The `guidelines.md` recommends `react-native-serial-transport`. This will involve native Android development (Java/Kotlin) to handle USB permissions (`android.permission.USB_PERMISSION`, `android.hardware.usb.host`), device discovery, and data passthrough to the JavaScript layer via JSI. |
| - FOC-Stim (TCP) | `PySide6.QtNetwork.QTcpSocket` | **High Feasibility** | React Native has mature libraries for TCP communication (e.g., `react-native-tcp-socket`). This is a straightforward port. |
| - NeoStim (Serial) | `PySide6.QtSerialPort` | **Medium-High Feasibility** | Same as FOC-Stim Serial. The custom binary protocol will need to be implemented in TypeScript. |
| - Audio-based Devices | `sounddevice` library | **High Feasibility** | Audio generation can be achieved in React Native using libraries like `react-native-audio-recorder-player` or by generating a WAV file and playing it. For real-time streaming, a more advanced setup with a library that provides a raw audio buffer callback would be needed. |
| **3. Media Synchronization** | | | |
| - Local Funscript Loading | Python `os`, `pathlib`, `zipfile` | **Medium-Low Feasibility (as-is)** | Direct filesystem scanning as done on desktop is not idiomatic or practical on Android due to scoped storage. The mobile app should use the Storage Access Framework (`react-native-document-picker`) for users to explicitly select media files and funscript directories. This is a significant UX change from the desktop version. |
| - Media Player Sync | Sockets (TCP/UDP/WebSocket) | **High Feasibility** | The network-based integrations with players like VLC, Kodi, HereSphere, and MPC-HC can be ported directly using React Native's networking APIs. This is a strong feature for mobile. |
| **4. Firmware Flashing** | `stm32loader` library over serial | **Low Feasibility** | Flashing firmware from a mobile device is a complex and risky operation. It involves low-level serial communication and direct hardware control that is not well-suited for a React Native application. This feature is a prime candidate for **deprioritization** in the initial mobile release. |
| **5. UI & Configuration** | PySide6 (Qt) | **High Feasibility (Re-implementation)** | The entire UI will need to be rebuilt using React Native components. The `guidelines.md` specifies using Expo Router for navigation and React Native 0.83+ features. The existing UI structure (tabs, dialogs) provides a good blueprint for the mobile UX, but it must be adapted for smaller screens. |
| **6. Pattern Generation** | `QTimer` driving updates | **High Feasibility** | This can be implemented in JavaScript using `setInterval` or `requestAnimationFrame`. The pattern logic itself is simple math and can be ported directly. |
| **7. File/Script Tools** | Funscript/Simfile conversion and decomposition | **Medium Feasibility** | The conversion logic is pure Python and can be ported. However, the UI for file selection and management will need to be adapted for Android's filesystem constraints. This could be considered a secondary feature for the mobile app. |
| **8. Teleplot Data Output** | UDP-based output for real-time plotting in VS Code. | **Low Feasibility (Not for Mobile)** | As per user feedback, Teleplot functionality, which is primarily for debugging and advanced analysis via external tools like VS Code, is not suitable or feasible for direct porting to the mobile application. While the underlying telemetry data from FOC-Stim could eventually be used for in-app visualizations, the Teleplot *mechanism* itself is out of scope for mobile. |
| **9. T-Code Servers** | `QTcpServer`, `QUdpSocket`, `QWebSocketServer` | **Medium-High Feasibility** | Native modules or JavaScript libraries exist to create servers on mobile, but this is an unusual feature for a mobile app. It may have implications for battery life and background processing. Its priority should be evaluated. It's likely more important to *receive* commands (e.g., from Buttplug) than to *host* servers. |

## Summary of Key Mobile Challenges

1.  **USB-Serial Communication**: This is the biggest technical hurdle. It requires a well-maintained native module and careful handling of Android permissions and the USB host lifecycle. The recommended `react-native-serial-transport` should be investigated thoroughly.
2.  **Filesystem Access**: The desktop app's model of scanning user-defined directories for media and scripts is not directly translatable to Android. The UX must be redesigned around the Storage Access Framework, where the user explicitly grants access to files/folders.
3.  **Firmware Flashing**: This is a high-risk, low-reward feature for a mobile app. It's strongly recommended to **exclude this from the initial scope**. The desktop application can remain the designated tool for firmware updates.
4.  **Background Processing**: For the command loops to run reliably (especially for FOC-Stim/NeoStim), the app may need to run as a foreground service to prevent Android from killing it. This has UX implications (a persistent notification is required). `react-native-worklets-core` or similar may be needed to ensure the JS loop is not throttled.
5.  **Python to TypeScript/JS Porting**: While the logic is portable, a direct line-by-line translation is not always ideal. The `stim_math` library should be ported idiomatically to TypeScript to leverage its type system, which aligns with the project guidelines.
